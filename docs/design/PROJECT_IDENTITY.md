# Project Identity

## Core Question

What should this product feel like?

Portfolio should feel like a calm, modern, professional engineering product created by a Frontend Developer who understands product goals, user perception, technical quality, and maintainability.

It should not feel like a template, a resume export, a playground, or a visual experiment.

## Desired Feeling

The product should feel:

- modern;
- calm;
- professional;
- technological;
- minimalistic;
- accurate;
- confident;
- not overloaded.

## Personality

Portfolio should communicate quiet confidence. The interface should not shout for attention. It should earn attention through clarity, rhythm, details, and consistency.

The product voice should be:

- direct;
- thoughtful;
- concise;
- human;
- technically credible.

## What the Product Should Communicate

The viewer should feel that the developer:

- can structure work before writing code;
- understands frontend as engineering, not only visual assembly;
- can communicate decisions clearly;
- respects maintainability and accessibility;
- knows how to use modern tools without overcomplicating the product;
- can collaborate effectively with AI while keeping technical ownership.

## What the Product Must Avoid

Portfolio must avoid:

- looking like a generic online resume;
- excessive animations;
- overloaded layouts;
- decorative complexity;
- vague self-promotion;
- unnecessary sections before the MVP is complete;
- technology choices without product purpose;
- visual noise that makes scanning harder.

## Recruiter Perception

For recruiters, the product should quickly answer:

- who this candidate is;
- what role they fit;
- what value they can bring;
- why they are worth forwarding to a technical interview.

## Engineering Perception

For technical reviewers, the product should demonstrate:

- clear architecture;
- clean component boundaries;
- restrained dependency choices;
- thoughtful UI implementation;
- accessible frontend practices;
- production readiness.

## AI-Assisted Development Identity

AI is part of the project's development process, but it does not replace engineering judgment.

The project should show that AI can be used to:

- speed up implementation;
- improve documentation;
- explore alternatives;
- support review and iteration.

The developer remains responsible for:

- architecture;
- product decisions;
- code quality;
- final review;
- deployment readiness.

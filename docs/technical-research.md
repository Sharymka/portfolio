## 0. Как читать этот документ

Портфолио оценивают три разные аудитории одновременно:

- **Рекрутер** — красиво, быстро грузится, понятно за 10 секунд, работает на телефоне.
- **Frontend-разработчик** — читает код, смотрит, не «резюме-driven development» ли это (технологии ради галочки в CV).
- **Технический интервьюер** — смотрит, есть ли **инженерное суждение**: почему выбрано именно это, а не модное. Осознанный отказ от технологии — такой же сильный сигнал, как её грамотное использование.

Поэтому ключевой принцип ниже: **каждая технология в проекте должна быть либо необходима продукту, либо демонстрировать конкретный навык в изолированном, оправданном месте** — а не просто присутствовать в package.json.

## 1. Версии Next.js и React

**Next.js: 16.2.x (текущий стабильный LTS-канал)**

- Next.js 16 — актуальная стабильная линейка на июль 2026 (16.2.10 LTS, есть препрелиз 16.3 с доп. фичами навигации, но брать стабильную ветку 16.2.x).
- Требования Next.js 16: Node.js ≥ 20.9 (18 больше не поддерживается), TypeScript ≥ 5.1.
- Важные изменения по сравнению с Next 14/15, которые нужно сразу закладывать в архитектуру:
  - Turbopack используется по умолчанию (`next dev`/`next build`) — Webpack больше не дефолт.
  - Синхронный доступ к `cookies()`, `headers()`, `params`, `searchParams` полностью удалён — везде `await`.
  - `fetch()` больше не кэшируется по умолчанию (это уже с Next 15) — кэширование нужно объявлять явно.
  - React Compiler дошёл до статуса stable, но **не включён по умолчанию** — включается флагом `reactCompiler: true` в `next.config.ts`, требует Babel-плагин, увеличивает время сборки. Стоит включить осознанно и явно, а не молча.

**React: 19.2.x**

- Next.js 16 требует `react ^19.0.0` (поддержка React 18.2 тоже есть, но брать 19 — React 19 стабилен с декабря 2024, последний патч на июль 2026 — 19.2.7). React 19 даёт Server Components, Server Actions, `useActionState`, `useOptimistic` «из коробки» — это прямое попадание в пункт «оптимистичные обновления» из вашего стека, без Redux.

Источники: [Next.js Blog](https://nextjs.org/blog), [Next.js Upgrading v16](https://nextjs.org/docs/app/guides/upgrading/version-16), [React Versions](https://react.dev/versions), [endoflife.date/nextjs](https://endoflife.date/nextjs), [endoflife.date/react](https://endoflife.date/react).

## 2–17. Разбор по каждому пункту стека

### 2.1 TypeScript, JavaScript

TypeScript — да, строго. JavaScript в вашем списке избыточен как отдельный пункт: в TS-проекте JS-файлов не будет в принципе, кроме конфигов, где это неизбежно. Не позиционировать как отдельную технологию, это часть TS-стека.

### 2.2 SCSS Modules vs альтернативы

**Оставляем SCSS Modules** — это ваш заявленный сильный навык, и для портфолио, которое должны читать frontend-разработчики, именно ручная работа со стилями (не Tailwind-классы) лучше показывает владение CSS: продуманные BEM-подобные имена, миксины, токены через CSS custom properties, адаптивность через media/container queries.

Альтернатива — Tailwind CSS — сейчас более «модный» индустриальный стандарт, но:

- Не входит в ваш заявленный стек (не показывает то, что вы хотите показать).
- Скрывает реальное владение CSS за утилитарными классами — интервьюеру сложнее оценить CSS-скилл.

**Решение:** SCSS Modules + CSS custom properties как design tokens (цвета/spacing/typography в `:root`, тема light/dark через `data-theme`). Это даёт полноценную мини design-систему без лишней библиотеки.

### 2.3 Redux Toolkit и RTK Query — нужны ли без сложного backend?

**Нет, не нужны как основной слой стейт-менеджмента.** Это самый спорный пункт вашего списка, разберу подробно.

**За (аргументы «оставить»):**

- Указаны в вашем опыте — хочется показать.
- RTK Query действительно умеет кэширование/инвалидацию/оптимистичные обновления «из коробки».

**Против:**

- У портфолио нет сложного backend и почти нет **разделяемого клиентского состояния** между удалёнными фичами — контент кейсов статический, форма обратной связи — одна изолированная мутация.
- Redux Toolkit — это Provider, store, слайсы, typed hooks — существенная церемония ради 1-2 бит состояния (тема, локаль, мобильное меню). Технический интервьюер увидит в этом resume-driven development, а не инженерное решение.
- Next.js 16 + React 19 закрывают ровно то, что вы хотели показать словами «кеширование», «инвалидация», «оптимистичные обновления» нативно: Server Actions, `useActionState`, `useOptimistic`, кэш `fetch()`, `revalidateTag`/`revalidatePath`.

**Решение (подтверждено):** Redux Toolkit и RTK Query в проект не включаются вообще — ни как основной стейт-менеджер, ни как изолированная опциональная фича (live-виджет с GitHub-статистикой рассматривался и отклонён). Локальный UI-стейт закрывается React `useState`/`useReducer`/context, где действительно нужно (тема, язык — уже решается next-intl без своего стейт-менеджера).

Это тот случай, где **явный отказ от технологии из вашего же списка — сильный сигнал** для технического интервью: «я умею X, но не притащил его туда, где он не нужен». Причину стоит явно зафиксировать в README проекта.

Источники: [Redux Toolkit RTK Query Comparison](https://redux-toolkit.js.org/rtk-query/comparison), обсуждение сообщества по выбору между Zustand/RTK Query/TanStack Query для небольших приложений.

### 2.4 Где хранить данные проектов (кейсов)

Варианты и почему не CMS:

- **Headless CMS (Contentful/Sanity и т.п.)** — избыточно. Добавляет внешнюю зависимость, требует поддержки бесплатного тарифа, для одного автора без сторонних редакторов контента не даёт выгоды, а добавляет риск (аккаунт, лимиты, downtime стороннего сервиса) и площадь для вопросов на интервью не по вашей теме.
- **Только TypeScript-объекты** — хорошо для строго структурированных полей (заголовок, стек, ссылки, метрики), но неудобно для длинного форматированного текста кейса (заголовки, списки, изображения по ходу текста).
- **Только JSON** — та же проблема, плюс теряется типобезопасность без доп. слоя.
- **MDX** — хорош для форматированного текста и встраивания React-компонентов прямо в кейс (например, интерактивное сравнение "до/после"), но без схемы легко рассинхронизировать поля между кейсами.

**Рекомендация — гибрид:** структурированные метаданные (slug, роль, стек, даты, ссылки, обложка) описываются Zod-схемой и валидируются на этапе сборки; тело кейса — MDX. Технически это удобно собрать через **Velite** (легкий, активно поддерживаемый инструмент, генерирует типобезопасный content layer из MDX/YAML/JSON по Zod-схемам; пришёл на смену заброшенному Contentlayer). Итог: `content/projects/*.mdx` с frontmatter, провалидированным Zod → автогенерированные TS-типы → статическая генерация страниц.

**Согласовано с design-research:** UX-исследование (`portfolio-design-research.md`, п. 3) настаивает, что каждый кейс должен открываться блоком «роль → цель → результат» с метриками — до деталей реализации. Это значит, Zod-схема кейса должна с самого начала содержать **обязательные структурированные поля обзора**, а не просто произвольный MDX-текст:

```ts
overview: z.object({
  kind: z.enum(['business', 'learning']), // тип кейса — влияет на framing result
  role: z.string(),
  goal: z.string(),
  result: z.string(), // business: "конверсия +34%"; learning: "решена проблема X / освоен Y"
  metrics: z.array(z.object({ label: z.string(), value: z.string() })).optional(), // опционально, не обязательно для learning-кейсов
});
```

Поле `kind` явно фиксирует тип кейса, чтобы UI и AI-чат не пытались выдумать бизнес-метрику там, где её не было — честность важнее единообразия подачи. Это поле рендерится как первый блок страницы кейса (`Project Overview`), а само MDX-тело идёт ниже как «как это было сделано». Тот же `overview`-объект удобно переиспользовать как источник для карточки кейса на главной и как часть контекста AI-чата (п. 2.16) — один источник правды на три места использования.

Это прямое попадание в ваш стек (Zod уже заявлен) и хороший повод показать типобезопасный content pipeline — сильный сигнал для интервьюера.

Источники: [Velite](https://velite.js.org/guide/introduction), обсуждения о прекращении поддержки Contentlayer.

### 2.5 Динамические страницы кейсов

`app/[locale]/projects/[slug]/page.tsx`:

- `generateStaticParams()` по списку slug'ов из content layer → полностью статическая генерация (SSG) на этапе билда, ISR не нужен, так как контент обновляется вами, а не внешними пользователями.
- `generateMetadata()` для SEO/OG per-кейс.
- `notFound()` для несуществующего slug.

Это стандартный, скучный (в хорошем смысле) паттерн Next.js App Router — ровно то, что должен увидеть интервьюер: не изобретён велосипед там, где фреймворк уже даёт решение.

**Уточнение по объёму (решено):** на первом запуске — **3-5 кейсов**, по рекомендации design-research. Фильтрация/поиск по стеку/тегам (`features/project-filter`) при таком количестве всё ещё не нужна — простая статическая сетка карточек полностью закрывает задачу. Фичу-фильтр стоит закладывать в бэклог, а не в MVP: добавить, если число кейсов вырастет за 6-8 и навигация по простому списку станет неудобной.

**Оговорка (озвучена вами):** часть из этих 3-5 кейсов — учебные/пет-проекты, не работа с реальным бизнесом, поэтому классических бизнес-метрик («конверсия +34%») у них может не быть. Это не повод их исключать, но `overview`-блок (см. 2.4) должен честно адаптировать framing под тип кейса — учебный кейс продаёт себя через **техническую сложность и то, что было решено/освоено**, а не притворяется бизнес-результатом. Ложная метрика хуже, чем её отсутствие — тот же принцип честности, что и для AI-чата (2.16, риск 10 в 3.6).

### 2.6 Нужна ли форма обратной связи и как без backend

Да, форма обратной связи нужна — это ожидаемый recruiter-facing элемент портфолио. Backend отдельно поднимать не нужно:

- **Server Action** (React 19 / Next.js) принимает `FormData`, валидирует Zod-схемой на сервере (не только на клиенте — иначе валидация бутафорская), вызывает **Resend** (или аналог) для отправки письма вам на почту.
- React Hook Form на клиенте — для UX (мгновенная валидация, состояния полей), Zod-схема **одна и та же** переиспользуется и на клиенте (RHF resolver), и на сервере (Server Action) — это как раз показывает грамотное разделение UX-валидации и security-валидации.
- Защита от спама по минимуму: honeypot-поле + `useActionState` для пендинг/ошибок состояния, при желании — простой rate-limit (например, через Vercel KV/Upstash с щедрым бесплатным тarifом) — опционально, не обязательно на старте.

Это не «полноценный backend» — это одна serverless-функция под капотом Next.js, что и требовалось.

Источники: [Next.js Forms guide](https://nextjs.org/docs/app/guides/forms), обзор паттернов contact-form 2026 (Server Action → Resend как наиболее распространённый паттерн).

### 2.7 Библиотека анимаций — нужна ли

**Решение (подтверждено):** анимации — приоритет продукта, хотите разнообразные и заметные, не точечные. Это меняет баланс по сравнению с обычной рекомендацией «минимум анимаций» — но и повышает требования к дисциплине, чтобы разнообразие не превратилось в хаос и не убило производительность.

**Motion** (бывший Framer Motion) остаётся де-факто стандартом для React в 2026, хорошо работает с Server Components/App Router — берём его как основной инструмент.

Как совместить «разнообразные анимации» с «production-ready, без проседания метрик»:

- **Слоями сложности**, а не везде одинаково тяжело:
  - Hero и переходы между крупными секциями — заметные, «премиальные» анимации (появление, параллакс, layout-анимации Motion).
  - Карточки кейсов, списки — scroll-triggered reveal (`whileInView`), staggered-появление элементов.
  - Микро-интеракции (hover, focus, active) — CSS `transition`/`@starting-style`, не грузить Motion ради каждой кнопки.
  - Переходы между страницами — рассмотреть нативный **View Transitions API** (Next.js имеет экспериментальную поддержку) как более дешёвую альтернативу Motion для full-page transitions.
- **Обязательные ограничители**, чтобы «разнообразие» не стало антипаттерном для интервьюера:
  - Полное уважение `prefers-reduced-motion` — все нетривиальные анимации должны иметь урезанный/выключенный вариант.
  - Lighthouse/Core Web Vitals бюджет в CI (см. 2.15) — если анимации проседают по INP/CLS, это должно быть видно до мержа, а не после деплоя.
  - Анимации не должны блокировать доступность: интерактивные элементы должны быть управляемы с клавиатуры и достижимы скринридером независимо от состояния анимации.
  - Не анимировать текст кейсов/контента, критичного для чтения (можно анимировать контейнер один раз при входе, но не сам текст построчно) — иначе это будет мешать рекрутеру, а не впечатлять.

Такой подход даёт визуальное разнообразие, которое вы хотите, но остаётся объяснимым решением, а не «анимация ради анимации» — что тоже хороший ответ на вопрос технического интервьюера «а почему тут анимация».

Источники: [motion.dev](https://motion.dev/), обзоры React animation libraries 2026.

### 2.8 Интернационализация (RU/EN)

Да, нужна — и вот почему это не «фича ради фичи»:

1. Прямая демонстрация заявленного навыка (next-intl указан в стеке).
2. Практическая польза: вы, скорее всего, откликаетесь и на русскоязычные, и на международные позиции — двуязычное портфолио полезно продукту, а не только резюме.

**next-intl** — по данным на июль 2026 (v4.13.x) — стандартный выбор для App Router: работает в Server Components без костылей, встроенный роутинг локалей через middleware, типобезопасные ключи переводов.

**Решение (подтверждено): поэтапно, английский первым.** Это значит:

- Роутинг локалей (`/en/...`, `/ru/...`), middleware, структура `messages/en.json` + `messages/ru.json` закладываются **сразу, с первого этапа** — переключение архитектуры на i18n постфактум значительно дороже, чем сразу спроектировать под два языка.
- Реальный контент (UI-строки, кейсы, AI-контекст для чата) — сначала только на английском, `ru.json`/RU-контент кейсов держится частично пустым или с explicit TODO до отдельного этапа перевода.
- Content-схема кейсов (п. 2.4) должна с самого начала поддерживать частичную локализацию (например, каждое MDX-поле — по языкам, но не блокировать билд, если RU ещё не заполнен) — иначе на этапе добавления русского понадобится рефакторинг схемы.

Риск, который нужно держать в голове (см. раздел рисков): двуязычный контент кейсов в итоге всё равно **удваивает объём текста на поддержку** — при поэтапном подходе это просто растянуто во времени, а не устранено.

Источники: [next-intl.dev](https://next-intl.dev/docs/getting-started/app-router), npm next-intl.

### 2.9 Стили и дизайн-система

SCSS Modules (см. 2.2) + токены на CSS custom properties + небольшой `shared/ui` набор примитивов (Button, Card, Typography, Badge, Container, Section — они уже частично видны в текущей структуре репозитория). Дизайн-система не должна быть избыточной: 8-12 переиспользуемых примитивов с чёткими вариантами — этого достаточно для портфолио, полноценная библиотека токенов уровня продуктовой команды не нужна.

**Конкретные значения токенов — подтверждено в [`visual-style-research.md`](./visual-style-research.md):**

- **Палитра:** тёмная тема (основная) — фон `#0F172A`/`#11172A` (slate/navy), текст `#DFE5EC`/`#E5E5E5`, акцент — teal-градиент `#2DD4BF → #22D3EE`, растворяющийся в фон, а не плоская заливка. Светлая тема — фон off-white `#F4F4F4`, текст `#1B1B1B`–`#2E2E2E`, тот же акцентный teal (бренд-цвет не теряется при переключении темы).
- **Типографика:** Space Grotesk (заголовки) + Inter (текст) + JetBrains Mono (акцент — даты/теги стека/лейблы в кейсах), через `next/font`, модульная шкала (коэффициент ~1.25–1.333), минимум 16px тело текста, WCAG AA контраст.
- Оба набора значений (light/dark) прокидываются через `data-theme` на `:root`, как и было решено в этом документе изначально — visual-research лишь подставил в этот механизм конкретные hex-значения, архитектурно ничего не меняет.

Полный разбор трендов, референсов, альтернативной палитры и обоснования — в `visual-style-research.md`, здесь только итоговые значения для реализации.

### 2.10 Архитектура: полноценный FSD, упрощённый FSD, или другое

**Рекомендация — урезанный FSD, не полный.**

Общая критика FSD для небольших проектов (подтверждается индустриальными обсуждениями): полный набор слоёв (`app / processes / pages / widgets / features / entities / shared`) на проекте с одним разработчиком и без сложных пользовательских сценариев приводит к избыточному количеству папок ради самой методологии, а не ради читаемости.

Что оставить:

- `app/` — в Next.js App Router эта роль частично уже занята нативной папкой `app/` фреймворка (роутинг). Слой FSD `app` здесь — это providers, глобальные стили, инициализация (то, что не про роутинг).
- `widgets/` — крупные составные блоки страниц (Hero, ProjectCard grid, ContactForm) — уже используется в текущем репозитории, оставляем.
- `entities/` — доменные модели (Project, SkillCategory) — оставляем, но не плодить entity там, где нет домена (не заводить `entities/button`).
- `features/` — только там, где есть настоящая пользовательская _интеракция_ с бизнес-логикой (переключение языка, отправка формы, фильтрация кейсов по стеку). Если такой логики нет — не создавать пустые features-папки.
- `shared/` — ui-kit, утилиты, типы — оставляем, уже есть в репозитории.
- **`processes/` — убрать.** Этот слой уже считается устаревшим в самой FSD (в методологии от 2.0 рекомендуют не использовать), а для портфолио без сложных кросс-фичевых сценариев (типа многошаговой авторизации) он точно не нужен.
- Слой `pages/` из классического FSD в Next.js App Router не нужен как отдельная папка — его роль выполняют `app/[locale]/.../page.tsx`, которые импортируют `widgets`.

Итог: `app/ (Next.js routing) + widgets + entities + features (точечно) + shared` — это осознанно урезанная FSD, а не «взяли всё по методичке». В README стоит явно объяснить, почему `processes` и отдельный `pages`-слой не используются — это тоже сигнал инженерного суждения.

Источники: [feature-sliced.design](https://feature-sliced.design/), критические разборы FSD для небольших проектов.

### 2.11 Какие тесты действительно полезны

Не нужно гнаться за 100% покрытием — это разоблачается на первом же вопросе интервьюера "а зачем вы тестируете эту чистую verstka-обёртку". Полезны:

1. **Unit-тесты** (Vitest) — чистые функции: форматирование дат, Zod-схемы валидации формы, утилиты выборки/фильтрации кейсов.
2. **Компонентные тесты** (Vitest + React Testing Library) — компоненты с реальной логикой: форма обратной связи (валидация, состояния submit/error/success), переключатель языка, фильтр списка кейсов. Не тестировать чисто презентационные компоненты (Badge, Typography) через RTL — это тестирует React/CSS, а не ваш код.
3. **Accessibility-проверки** — `vitest-axe`/`jest-axe` на ключевых экранах (главная, страница кейса, форма) — дешёвый, но заметный для интервьюера штрих.
4. **1-2 E2E-теста** (Playwright) на самый критичный сквозной путь: открыть кейс → переключить язык → отправить форму. Не десятки сценариев — именно демонстрация, что вы умеете писать E2E, а не полноценный regression-suite, который тут просто не нужен по объёму продукта.

Источники: обзор "Component Testing in 2026" (сочетание Vitest + RTL + Playwright как стандартного стека).

### 2.12 Нужен ли Storybook, и для каких компонентов

Да, но **точечно**, не для всего дерева компонентов:

- Обязательно: `shared/ui` — переиспользуемые примитивы (Button, Card, Badge, Typography, Container, Section, Textarea, Separator) — они уже частично есть в репозитории, и это именно то, ради чего Storybook существует: изолированная разработка и документация UI-кита.
- Опционально: 1-2 крупных widget (например, `ProjectCard`, `ContactForm`) с разными состояниями (loading/error/success) — хорошо показывает работу со стейтом компонента в изоляции.
- Не нужно заводить story на страницы целиком (`page.tsx`) — это не единица переиспользования, Storybook там не даёт пользы.

В 2026 Storybook 9 официально интегрируется с Vitest (addon на замену устаревшему test-runner): те же story можно гонять как interaction-тесты и a11y-тесты через Vitest browser mode — это позволяет не дублировать RTL-тесты и Storybook-тесты отдельно, а получить два применения из одного набора историй. Задеплоенный статический Storybook (Chromatic или GitHub Pages) — это ещё и отдельная "визитка" для frontend-рецензента, которую можно приложить в резюме отдельной ссылкой.

Источники: [Storybook + Vitest](https://storybook.js.org/blog/component-test-with-storybook-and-vitest/), [Storybook Writing Tests](https://storybook.js.org/docs/writing-tests).

### 2.13 SEO, accessibility, производительность, аналитика

- **SEO**: Next.js Metadata API (`generateMetadata`) на каждой странице/кейсе, `sitemap.ts`/`robots.ts` (нативные файловые конвенции App Router), JSON-LD (`Person`, `CreativeWork`/`Project`) для rich snippets, канонические URL с учётом локалей (`hreflang` через next-intl).
- **Accessibility**: семантический HTML, `eslint-plugin-jsx-a11y` в CI, ручная проверка клавиатурной навигации и фокуса, контраст цветов по WCAG AA как минимум, `prefers-reduced-motion` (см. 2.7), автоматизированные axe-проверки (см. 2.11).
- **Производительность**: `next/image` (обязательно, не сырые `<img>`), `next/font` для шрифтов без layout shift, анализ бандла (`@next/bundle-analyzer`), Core Web Vitals.
- **Аналитика**: Vercel Analytics + Speed Insights (бесплатный тариф, нулевая конфигурация при деплое на Vercel) — достаточно для портфолио; полноценный Google Analytics/Plausible — опционально, не обязательно на старте (лишний cookie-consent UX ради портфолио того не стоит).

### 2.14 Деплой

**Vercel.** Next.js разрабатывается той же командой, деплой буквально "подключить репозиторий" — zero-config, автоматические preview-деплои на каждый PR (что также хороший артефакт для демонстрации workflow технических рецензентов), бесплатного тарифа с запасом хватает для портфолио-трафика. Альтернативы (Cloudflare Pages, Netlify) обсуждать нет смысла — они добавляют трение (adapter-слои для part фич Next.js) без выгоды именно для этого проекта.

### 2.15 Проверки качества в CI

GitHub Actions, на каждый PR:

1. Установка зависимостей (с кэшем).
2. Typecheck — `tsc --noEmit`.
3. Lint — ESLint (или Biome, см. ниже) + `eslint-plugin-jsx-a11y`.
4. Формат — Prettier check (или Biome format check).
5. Unit/компонентные тесты — Vitest (+ coverage порог, но не 100%).
6. Сборка — `next build` (ловит SSR/типовые ошибки, которые dev-режим прощает).
7. Storybook build (валидирует, что UI-кит собирается) — опционально Chromatic для визуального regression.
8. Lighthouse CI с бюджетом на Core Web Vitals — опционально, но хороший штрих для "производительность" пункта.

**Отдельный вопрос — Biome vs ESLint+Prettier.** По состоянию на 2026: Biome в 10-25 раз быстрее и заменяет оба инструмента одним конфигом, но покрывает не все нишевые правила, а `eslint-plugin-jsx-a11y` и `eslint-plugin-next` пока остаются на ESLint. Рекомендация для соло-разработчицы: **не усложнять** — взять привычный ESLint (`eslint-config-next`) + Prettier, это mainstream-выбор, не требует объяснений на интервью и не создаёт риск "непривычный инструмент, о котором интервьюер начнёт расспрашивать вместо основного кода". Biome можно оставить опциональным upgrade потом.

Источники: [Biome vs ESLint 2026 обзоры](https://www.pkgpulse.com/guides/biome-vs-eslint-vs-oxlint-2026).

### 2.16 AI-чат «спросите обо мне» — новая фича, требует отдельного backend-узла

Это единственное место в проекте, где полноценно оправдан собственный serverless backend — не потому что «модно», а потому что вызов LLM API **обязан** идти через сервер: ключ провайдера нельзя класть в клиентский бандл, а сам вызов стоит денег и должен быть под контролем (лимиты, злоупотребления).

**Провайдер — решение подтверждено (2026-07-17): бесплатный тариф вместо платного.**

Изначальная рекомендация (Claude Haiku 4.5, $1/$5 за 1M токенов) остаётся самой дешёвой платной опцией, но вы подтвердили приоритет — «малой кровью» и без денежного риска, при этом не жертвуя тем, что чат остаётся вашим кодом (в отличие от no-code SaaS-виджетов вроде Chatbase — они были рассмотрены и отклонены отдельно: чужой продукт с вашим текстом внутри не даёт того инженерного сигнала, ради которого раздел 0 этого документа вообще написан).

|                                         | Google Gemini API                                                                                      | Groq                                                | Anthropic Claude API (отклонено)                                      |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------ | --------------------------------------------------- | --------------------------------------------------------------------- |
| Стоимость                               | **Бесплатно навсегда** на уровне проекта — не триальные кредиты                                        | **Бесплатно навсегда**, без системы кредитов вообще | Платно: Haiku 4.5 — $1/$5 за 1M токенов                               |
| Лимиты (модель `gemini-2.5-flash-lite`) | ~1000-1500 запросов/день, 15 RPM, 250K TPM, контекст 1M токенов                                        | 30 RPM, ~1K-14.4K запросов/день (зависит от модели) | Ограничено только бюджетом                                            |
| Карта нужна?                            | Нет                                                                                                    | Нет                                                 | Да                                                                    |
| Риск                                    | Нужен «чистый» Google Cloud проект без привязанного billing — иначе бесплатный тариф пропадает целиком | Организационный лимит, не обходится доп. ключами    | Прямой денежный риск при злоупотреблении (см. было в риске 7, п. 3.6) |
| SDK                                     | `@ai-sdk/google` через Vercel AI SDK — тот же паттерн стриминга                                        | `@ai-sdk/groq`, тот же SDK                          | `@ai-sdk/anthropic`, тот же SDK                                       |

**Решение:** Google Gemini API, модель **`gemini-2.5-flash-lite`**, без резервного провайдера (Groq рассматривался как fallback на случай исчерпания дневной квоты, но осознанно не берём его на старте — один провайдер = меньше кода и меньше конфигурации env-переменных, ровно то, что было запрошено; добавить Groq как fallback можно позже, если дневная квота Gemini окажется тесной на практике). Архитектура ниже не зависит от выбора провайдера — заменяется одним адаптером Vercel AI SDK.

**Границы чата — подтверждено: строго только вопросы о вас** (не общий ассистент). Это одновременно и продуктовое, и защитное решение:

- System prompt жёстко ограничивает тему («отвечай только на вопросы о профессиональном опыте, кейсах и навыках Светланы; на всё остальное — вежливый отказ»).
- Меньше площади для злоупотреблений (никто не использует ваш чат как бесплатный ChatGPT за ваш счёт).

**Источник знаний о вас — без RAG, без векторной БД.** При объёме контента портфолио (несколько кейсов, био, стек) достаточно **собрать весь контекст в system prompt** прямо из того же content layer, что и кейсы (п. 2.4, Velite/MDX + Zod, включая структурированный `overview` каждого кейса) — один источник правды, а не отдельная база знаний. Полноценный RAG (embeddings, vector store) был бы явной overengineering-ошибкой на этом объёме контента — если контента станет заметно больше (десятки кейсов/статей), тогда можно пересмотреть.

**Дополнено на основе `portfolio-design-research.md` (продуктовые требования к чату, не только технические):**

1. **Строгий запрет на «додумывание».** System prompt должен явно требовать отвечать только на основе переданного контекста и **честно признавать, если ответа в контенте нет** («это лучше уточнить у меня напрямую») — вместо уверенно звучащей выдумки. Это критично: неправда, которую рекрутер потом опровергнет на интервью, вредит больше, чем отсутствие фичи.
2. **Стартовые подсказки-вопросы** в UI виджета (3-4 штуки, например «Какой опыт с Next.js?», «Почему стоит пригласить на интервью?») — снижают барьер входа, часть UI-контракта `widgets/ai-chat`, не только промпт-инжиниринга.
3. **Размещение (подтверждено) — финальная секция страницы («Ask My AI»), спокойный статичный блок, не floating-виджет.** Рекрутер сначала читает факты (Hero → кейсы → как думаете), чат — заключительный интерактивный шаг, а не что-то, отвлекающее с первой секунды. Это чуть снижает и площадь для случайных/шумовых запросов по сравнению с постоянно видимым плавающим чатом (хотя rate limiting всё равно обязателен, см. ниже). Визуальное оформление секции — см. новый раздел «Стиль AI-чата» в [`visual-style-research.md`](./visual-style-research.md).

**Архитектура:**

- `app/api/chat/route.ts` — Route Handler, вызывает LLM через **Vercel AI SDK** (`streamText`), стримит ответ клиенту (`result.toDataStreamResponse()`).
- Клиент — `widgets/ai-chat`, рендерится как финальная секция страницы (не sticky/floating), использует `useChat` из `@ai-sdk/react` (стриминг, состояние сообщений, ошибки — из коробки, без своего Redux/стейт-менеджера).
- API-ключ — только в серверных env-переменных (Vercel Environment Variables), никогда не попадает в клиентский бандл.

**Защита от злоупотреблений — пересмотрено под бесплатный провайдер.** С платным Claude Haiku rate limiting был обязателен именно из-за прямого денежного риска (см. было в риске 7, п. 3.6). На бесплатном тарифе Gemini денежного риска нет — риск теперь другой: **исчерпание общей дневной квоты проекта** (одной на всех посетителей сайта), после чего чат перестаёт отвечать всем до сброса лимита в полночь по Тихоокеанскому времени. Это меняет уровень защиты с «обязательно и заранее» на тот же режим, что и опциональный rate-limit формы (п. 2.6) — простая мера, а не отдельная инфраструктура:

1. **Простой in-memory rate limit** прямо в `app/api/chat/route.ts` (например, `Map<ip, timestamps[]>` с ограничением ~5-10 запросов в минуту на IP) — без Upstash/Redis на старте. Компромисс, который стоит держать в голове: на serverless/edge-рантайме память не гарантированно переживает между вызовами и не общая между инстансами, так что это «мягкая» защита от одного скрипт-кидди в цикле, а не строгая гарантия — для портфолио-трафика этого достаточно. Если станет тесно на практике — апгрейд до Upstash Redis (бесплатный тариф, 500K команд/месяц с запасом) добавляется без изменения остальной архитектуры.
2. Ограничение длины пользовательского ввода и `max_tokens` на ответ — остаётся обязательным (защищает не от денег, а от абсурдно длинных запросов/ответов и от расходования дневной квоты запросов одним пользователем).
3. Низкая `temperature`, чёткий system prompt, инструкция не раскрывать сам system prompt и не выполнять инструкции, присланные пользователем в сообщении (базовая защита от prompt injection) — не зависит от провайдера, остаётся как было.
4. Точка отказа: если дневная квота Gemini исчерпана — чат должен явно и красиво сообщать об этом («AI-ассистент временно недоступен, вернётся завтра» или ссылка на форму обратной связи), а не падать с ошибкой 500.

Источники: [Vercel AI SDK — Next.js App Router getting started](https://ai-sdk.dev/docs/getting-started/nextjs-app-router), [Gemini API Rate Limits](https://ai.google.dev/gemini-api/docs/rate-limits), [Groq Rate Limits](https://console.groq.com/docs/rate-limits), [Upstash Redis Pricing & Limits](https://upstash.com/docs/redis/overall/pricing), [AI SDK — Rate Limiting guide](https://ai-sdk.dev/docs/advanced/rate-limiting).

### 3.2 От чего предлагается отказаться

- **Redux Toolkit / RTK Query** — исключены полностью, включая ранее рассматриваемый вариант с изолированным live-виджетом (GitHub-статистика) — вы подтвердили, что он не нужен. Заменяется нативными возможностями React 19/Next.js 16 (Server Actions, `useActionState`, `useOptimistic`, `fetch`-кэш).
- **Headless CMS** — избыточная внешняя зависимость для контента одного автора.
- **RAG/векторная база для AI-чата** — при 3-5 кейсах и небольшом объёме био достаточно передавать контекст напрямую в system prompt; отдельная база знаний была бы overengineering на этом объёме.
- **Фильтр/поиск по кейсам** (`features/project-filter`) — не нужен при 3-5 кейсах на старте, простая статическая сетка достаточна; в бэклог, не в MVP.
- **Слой `processes` из классической FSD** — устарел даже в самой методологии.
- **Отдельный слой `pages` из FSD** — его роль уже выполняет `app/` роутинг Next.js.
- **JavaScript как отдельный пункт стека** — поглощается TypeScript.
- **Полноценная Google Analytics-аналитика с cookie-consent** на старте — не нужна ради портфолио, Vercel Analytics достаточно.

### 3.3 Технологии, оставленные опциональными

- **Biome** вместо ESLint+Prettier — можно внедрить позже как upgrade, не блокирует старт.
- **Playwright** визуальный regression / Chromatic для Storybook — приятный бонус, не критично.
- **Простой rate-limit** для формы обратной связи (Upstash/Vercel KV) — добавить при первых признаках спама, не заранее (в отличие от AI-чата, где rate limiting обязателен сразу, см. 2.16).
- **React Compiler** (`reactCompiler: true`) — включить осознанно после того, как приложение стабилизируется, замерив влияние на билд.
- **View Transitions API** для переходов между страницами — попробовать как более дешёвую альтернативу Motion для full-page transitions, если экспериментальная поддержка Next.js окажется достаточно стабильной.
- **RU-контент** (кейсы, UI-строки, AI-контекст) — весь i18n-каркас закладывается сразу, но фактическое наполнение русского языка — отдельный этап после английского.

### 3.4 Обоснование — см. разделы 2.1–2.15 выше, каждое решение разобрано отдельно с плюсами/минусами.

### 3.6 Возможные технические риски

1. **Удвоение контента из-за RU/EN** — каждый кейс нужно писать дважды; при отсутствии дисциплины часть контента может "отставать" на одном языке дольше другого. Митигировать: с самого начала закладывать i18n-схему валидации, которая требует оба языка одновременно (Zod `refine`, чтобы билд падал при неполном переводе).
2. **Соло-поддержка Storybook + Vitest + Playwright одновременно** — три тестовых инструмента для одного разработчика — риск, что часть тестов начнёт "гнить" (устаревать) без code review со стороны. Митигировать: держать периметр тестов маленьким и осознанным (см. 2.11-2.12), не гнаться за покрытием.
3. **FSD как псевдо-архитектура ради галочки** — если границы слоёв (`entities`/`features`/`widgets`) не будут реально соблюдаться (например, `widgets` начнёт напрямую тянуть API), это будет хуже, чем вообще не использовать FSD — интервьюер это заметит быстрее, чем отсутствие методологии.
4. **React Compiler + Turbopack** — относительно новая связка (React Compiler stable только в Next 16), возможны неочевидные баги на грани экосистемы; включать не по умолчанию, тестировать отдельно перед мержем.
5. **Resend / email-деплой формы** — бесплатные тарифы транзакционной почты имеют лимиты и требуют верификации домена; заложить время на настройку DNS (SPF/DKIM) до релиза формы.
6. **Velite** — новее и меньше community, чем Contentlayer был на пике; если пойдёт не так, fallback — writing собственного простого MDX-loader с `next-mdx-remote` + ручной Zod-валидацией — не критичный риск, но стоит держать в голове.
7. **AI-чат — стоимость и злоупотребления.** Публичный эндпоинт, вызывающий платный LLM API, без защиты может превратиться в неожиданный счёт (кто-то зациклит запросы или начнёт использовать чат не по назначению). Митигировать строго до релиза, не после: rate limiting на уровне Route Handler (п. 2.16), ограничение длины ввода/вывода, минимальный system-prompt guardrail против prompt injection.
8. **AI-чат — устаревание контекста.** System prompt собирается из content layer на момент сборки — при обновлении кейсов/резюме нужно не забывать, что и чат должен видеть актуальную версию (по факту — просто пересборка/редеплой, но это стоит проговорить как процесс, а не только как код).
9. **Насыщенные анимации vs. производительность.** Явный риск, если не держать бюджет: чем разнообразнее анимации, тем легче случайно просадить INP/CLS или получить укачивание у части пользователей. Митигировать Lighthouse-бюджетом в CI и обязательным `prefers-reduced-motion` (п. 2.7) — не постфактум, а с первой анимации.
10. **AI-чат — недостоверный ответ подрывает доверие сильнее, чем его отсутствие.** Это главный риск фичи по `portfolio-design-research.md`: если чат «додумает» несуществующий опыт, а рекрутер поймает несостыковку на интервью — эффект будет хуже, чем если бы фичи не было вообще. Митигировать строгим system-prompt guardrail («только из контекста, иначе честно — не знаю», см. 2.16) и обязательным ручным review содержимого `overview`-блоков перед тем, как они попадут в контекст чата.
11. **Готовность контента и «сила» учебных кейсов.** Тексты ещё не готовы (подтверждено вами), а часть из 3-5 кейсов — учебные проекты, честность которых зависит от того, насколько убедительно описана техническая сложность (см. `overview.kind` в 2.4). План реализации (3.7) не должен блокироваться на «дождаться всех текстов» — сайт должен корректно работать и с меньшим числом готовых кейсов, добавление остальных не требует изменений в архитектуре.

### 3.7 Поэтапный план реализации (высокоуровнево, без деталей задач)

1. **Фундамент**: инициализация Next.js 16 + TS + ESLint/Prettier + структура папок (урезанный FSD) + CI-скелет (typecheck/lint/build) + верификация деплоя пустой страницы на существующий Vercel-проект «Portfolio».
2. **Design tokens и `shared/ui`**: SCSS-токены, базовые примитивы, Storybook-обвязка.
3. **i18n-каркас**: next-intl, роутинг локалей, структура `messages/en.json` + `messages/ru.json` (RU — пустой/TODO на этом этапе).
4. **Content layer**: Zod-схема кейса (включая `overview.kind: business | learning`), Velite-конфиг, 3-5 тестовых/плейсхолдерных MDX-кейсов на английском (реальные тексты — когда будут готовы, отдельным этапом, без изменения архитектуры).
5. **Страницы**: главная (Hero, уже частично есть), сетка кейсов (без фильтра, см. 3.2), динамическая страница кейса с `generateStaticParams`.
6. **Форма обратной связи**: RHF + Zod + Server Action + Resend, unit- и компонентные тесты.
7. **AI-чат «спросите обо мне»**: Route Handler + Vercel AI SDK, интеграция с выбранным LLM-провайдером, system prompt из content layer, rate limiting (Upstash) — до публичного включения на сайте обязательно закрыть пункт защиты от злоупотреблений (см. риск 7 в 3.6).
8. **Анимации**: Motion — hero/секции/scroll-reveal, CSS для микро-интеракций, `prefers-reduced-motion`, замер влияния на Lighthouse сразу по ходу добавления (не постфактум).
9. **Полировка**: SEO-метаданные, sitemap/robots, a11y-проход, Lighthouse-бюджет в CI.
10. **Тестовый периметр**: unit + компонентные тесты, 1-2 Playwright smoke-сценария (включая happy path AI-чата), a11y-автотесты.
11. **Деплой и аналитика**: финальная проверка Vercel-деплоя, Vercel Analytics/Speed Insights, финальный CI со всеми проверками.
12. **Отдельный этап после релиза MVP**: наполнение русского контента (кейсы, UI-строки, RU-ветка AI-контекста) — без изменений в архитектуре, только контент.
